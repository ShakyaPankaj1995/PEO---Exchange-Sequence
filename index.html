01 — Context

What the system does,
and what it was missing

The production order application manages manufacturing jobs end-to-end — bill of materials, work authorisation, defect tracking, and a schedule of operations. The schedule is the operational backbone. Changing it, however, was a capability the system never formally supported.

The three operation types

Standard operations

The primary sequence

Numbered operations (e.g. 10, 20, 30...) that define the main production path. Each carries a work centre assignment, duration, and status. Together they constitute the baseline schedule against which all other operations are measured.

Parallel operations

Branch and end logic

Operations that run alongside standard work. Each parallel operation references a branch operation (where it begins in relation to the standard sequence) and an end operation (where it rejoins). For example: branches at standard op 30, ends at standard op 45.

Alternative operations

The contingency layer

Reserve operations activated when standard or parallel work cannot proceed. Like parallel operations, they carry branch and end references. They exist in the system in a dormant state — available to replace, but requiring a deliberate exchange action to become active.

The exchange sequence problem

When something goes wrong in production — a machine breaks down, a process yields defects — the operations manager needs to swap a range of scheduled operations for an alternative set. This is called an exchange sequence.

The complexity: the replacement is not operation-for-operation. A block of four standard operations (say, ops 05, 10, 15, 20) might be replaced by just two alternative operations (ops 50, 60). There is no 1:1 mapping. And a single standard block may have two or more alternative sequences competing to replace it.

↯ The system had no concept of a "sequence" as a named entity. There was no sequence ID. Operations only knew their individual branch and end operation references — a structural gap that cascaded into every design decision that followed.
→ The design challenge was therefore not just a UI problem. It required inventing a legible concept of "sequence" entirely within the presentation layer — using only the data that existed — without touching the backend model.
02 — Constraints

The walls of the room:
what the design could not change

Understanding which constraints were fixed versus flexible was the first design task. Three hard boundaries shaped every decision that followed.

System constraint

No sequence identifier in the backend

The data model had no field for sequence ID — and the backend team confirmed none could be introduced. Operations only carried branch operation and end operation references. Any grouping into "sequences" had to be inferred and displayed entirely by the UI layer.

Component constraint

One attribute per table cell in Fiori

The SAP Fiori / S/4HANA design system restricts table cells to a single data attribute per cell. A standard approach — showing both the affected operations and the alternative operations in adjacent columns of the same row — was architecturally blocked by this rule.

UX constraint

Action must be preceded by context

Early proposal placed the exchange sequence action in the page header — always accessible. This was dropped. Before initiating the exchange, the operator must see exactly which operations will be replaced by which alternatives. A header action offers no such visibility. Context must precede action.

Why these constraints actually helped

Constraints forced a more honest solution. Without the backend limitation, a designer might reach for a simple ID-based lookup. Without the Fiori cell rule, a flat table with multiple data points per row would have seemed sufficient. Neither would have served the operator well — both would have buried critical relationships in data rather than surfacing them.

The constraints pushed the design toward a hierarchical presentation model — one that makes relationships between sequence level and operation level explicit and browsable. The result is arguably more transparent than a simple table would have been.

03 — Options Explored

Two directions,
one deliberate choice

Before committing to a solution, two structural approaches were mapped and pressure-tested against the constraint set.

Option A — Header action

Dropped

Place an "Exchange Sequence" action button in the object page header — visible and accessible regardless of which tab the user is currently viewing. The action would open a dialog or side panel to configure the exchange.

This approach mirrors common SAP Fiori patterns where global object actions sit in the header toolbar, giving them maximum discoverability and consistent placement across sections.

Why it was dropped: The action has no meaning without context. An operator seeing "Exchange Sequence" in the header has no way to know which operations are involved, what they will be replaced with, or whether conditions are safe to proceed. The action requires prior comprehension — and the header provides none. Discoverability without context is a false benefit.

Option B — Dedicated section with inline data

Chosen

Introduce a dedicated "Exchange Sequence" section within the order schedule area. This section lists all available alternative sequences alongside their coverage — the affected standard operations and the alternative operations that would replace them.

The action to initiate an exchange lives inside this section, surfaced only after the operator has had full exposure to the data. Selection happens with a radio button (enforcing single-sequence selection). The initiation triggers a confirmation dialog.

Why it was chosen: The section provides a deliberate pre-decision environment. Operators see the full picture before they can act. The action is discovered in context — not abstracted to a global toolbar. This aligns with how experienced operators think: assess, then decide, then act.
04 — Solution

The designed flow,
step by step

The solution is a linear decision flow — discovery, exploration, decision, confirmation — each stage progressively revealing more detail before the irreversible action is taken.

1
Discover

Exchange sequence section — the list view

A new section within the Order Schedule tab presents all available alternative sequences in a tree table. At the parent level, each row represents a sequence — identified by its branch and end operation pair, and the number of affected standard operations. The operator can scan available alternatives at a glance without committing to any detail.

05 — Design Trade-offs

Challenge, constraint,
decision

Every design decision involves a trade. Making these explicit is what separates design documentation from design thinking.

Challenge Constraint Decision
No sequence ID exists in the backend data model. Operations only carry individual branch and end operation references. Backend team confirmed no new fields could be introduced to the data model. Use the branch + end operation pair as the surrogate sequence identifier at the parent row level. This is meaningful to operators already familiar with branch/end logic.
Multiple alternative sequences can cover the same standard operation range — they need to be distinguished and individually selectable. Fiori table cells support only one data attribute per cell. A flat table with one row per sequence and one cell containing all alternatives was not possible. Tree table component separates the two hierarchy levels: sequences at parent level, operations at child level. Each level carries only the data meaningful at that level.
The action must be irreversible once confirmed — the operator needs to be certain before proceeding. A single confirmation button with no preview does not meet the safety standard for an operational change at this scale. Two-stage confirmation: the dedicated section provides pre-decision context, the dialog provides a final holistic review of both sides of the exchange before the commit button is enabled.
An exchange cannot proceed if any standard operation in the affected sequence is currently in process on the shop floor. The system enforces this as a hard block — the backend will reject the action. The UI must communicate this clearly without simply letting the action fail silently. Status visibility at every level: in the popup, in the tree expansion, and in the dialog. The disabled state in the dialog is accompanied by an explanatory message strip — not just a greyed-out button.
Two or more alternative sequences may be valid for the same standard block — the operator must choose between them, not activate multiple. Checkboxes would imply multi-select is possible. The system enforces single-sequence selection. Radio button selection at the sequence level enforces the constraint at the interaction layer — the component itself communicates the rule before the user tries to break it.
The global header action placement (Option A) offered high discoverability but zero context. The UX principle that actions requiring significant judgment should not be divorced from the data that informs that judgment. Dropped Option A entirely. Context is not supplementary to this action — it is prerequisite to it. The dedicated section co-locates data and action in the same visual environment.

Error prevention

Blocked state = system safety

An In Process operation means the shop floor and system record would diverge if the exchange ran. The disabled button with message strip tells operators the exact blocker — not just a wall, but a direction.

Cognitive load

Staged disclosure under pressure

Each of the four steps reveals only the information relevant to that decision. When a disruption hits and time is short, the flow prevents overload — clarity under pressure is a safety property, not a UX nicety.

07 — Reflection

What this taught me about
designing within systems

Enterprise UX is not about designing the ideal interface. It is about designing the best possible interface given a fixed technological context, an established design system, and a user community who has already built workflows around the existing limitations.

The backend is a design constraint, not a design partner

The absence of a sequence ID was not a failure of backend design — it reflected a data model built for a different purpose. The UX layer's job was to bridge the gap between what the data model captures and what the operator needs to understand. Inventing "sequence" as a presentation concept, grounded in existing branch/end logic, was the only lever available.

Component constraints force hierarchy thinking

The one-attribute-per-cell Fiori rule felt like a limitation. In practice, it forced a two-level hierarchy that made the data more legible than a flat table would have been. The constraint eliminated a lazy solution and pushed toward a structure that actually reflects the conceptual model: sequences contain operations.

Discoverability is not the same as context

Option A's header placement would have scored well on a discoverability heuristic audit. It would have failed the moment an operator tried to use it without knowing what "Exchange Sequence" means in practice. Discoverability is a precondition for use — it is not sufficient. The placement needs to deliver the user to a state where they have what they need to decide.

If I had more time

I would have explored how the exchange sequence section behaves after an exchange has been performed — specifically, how the historical record of "this sequence was replaced by this alternative, on this date" would be surfaced. The current solution focuses on the initiation flow; the audit and reversal flows remain unresolved.
